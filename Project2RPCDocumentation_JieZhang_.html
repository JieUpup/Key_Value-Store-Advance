1. System setup and Running program

                                                

        This mini  server- client  program will serve as a key value store. It will be set up to allow client to communicate with the server and perform three basic operations:

1) PUT (key, value)                                        

2) GET (key)

3) DELETE(key)

       The project was developed by Java programming and RMI. The project looks into and leverages Java RMI for RPC communication  between client and server.

       For this project, unlimited threads will respond to multiple outstanding client requests at once.

        Also the servers can handle requests from multiple running instances of clients doing concurrent PUT, GET, and DELETE operations. Also, projects can handle mutual exclusions.                                                 

Note:

(1) This project is developed on jdk.17.0.1, running this program on any jdk that comes before jdk.17.0.1 may result in unknown issues. (e.g., unsupported syntax). If any issues have been met, please con- sider upgrading the jdk to a later version. To make it easy we provid- ed docker image based on the OpenJDK17 image.

                                                

(2)Run project in the terminal command

Compile the application

cd src

javac *.java

```

Need at least three terminals to run the application.

Terminal 1:

Start the registry server

rmiregistry

Terminal 2:

Start the RMI server``

java KeyValueStoreServer

Terminal 3:

Start the client to init 10 test key value pairs`

java KeyValueStoreClient

Then use the KeyValueStoreCmd to do more operations.

For examples:``

java KeyValueStoreCmd get key1

java KeyValueStoreCmd put someKey someValue

java KeyValueStoreCmd delete key1

                        

                                                

2. Project Technology Overview

                                                

           This project is a good way to learn how to implement the communi- cation between servers and clients using Remote Procedure Calls. For the first time, when I want to let the communication happen between server and client. I remembered my first homework and first project, I used Socket and TCP, UDP to connect them. For now, I need to understand the definition of RPC.

          Here is my basic knowledge about RPC:

          RPC(Remote Protocol Call) is a remote invoke agreement based on the C/S model. It can let the client send the invoke request and wait for the server response. It is a very basic request-response model in distributed systems. We can general the step like: local object sent the request  to local pc, and we don’t need to write the low level language and local pc will sent the request to the serve by the network, when the server received the parameter from the local pc, they will send back the result to the local client.

3. Project Technical Impression

                                                

        First I tried to understand the requirements for this project, then I di- vided the big task to small tasks and drew a project basic diagram at the beginning. I understand I need two main parts of this project, one is client and one is server. Then I tried to design an interface to avoid the inherited multiply relationship between classes.

          The big issue is which RPC framework I can use to achieve this project. For my limited knowledge, I know I can use RMI to handle this issue. But it is not a flexible way of comparing other tools.

            For this course project, RMI is lightweight and object oriented. So I still choose it to handle the communication issue between client and server.

           Here is my currently knowledge about RMI:

           We can consider the RMI as a Java programming language to let the RPC become more general. Because RPC can support the object commute . RMI is a more object oriented way to let objects connect with each other. It does not use SOAP to achieve it.

           RMI uses a Java remote message protocol to let the method be running in any operator system (but needs Java). Compared with RPC, RMI can use java garbage collection, object oriented. Also RMI uses stubs and skeletons to let remote objects communicate. Stub is an agency for remote objects. It has the same interfaces as the remote objects. Invoking a remote object to invoke the stub to achieve it if using RMI. The date transmission when using RMI is java objects, not XML data.                                        

        It’s also a good experience to use RPC and RMI to communicate between client and server. If I have enough time, I hope I can use other RPC frames to achieve this project such as snifty and swift.  Also, I hope there is a more general way to connect with an HTTP server.
